# Finite Fields and Elliptic Curves

This chapter covers the mathematical foundation underlying all of FROST:
finite field arithmetic and elliptic curve point operations on secp256k1.

## The Prime Field F_p

Ordinary arithmetic on integers gives you results that grow without bound.
Cryptography needs a bounded structure where addition, subtraction,
multiplication, and division all stay within a fixed set. A *finite field*
provides exactly this.

A prime field F_p is the set of integers {0, 1, 2, ..., p-1} with arithmetic
performed modulo p. The "modulo" operation wraps results back into the range:

    7 + 5 = 12   but   7 + 5 ≡ 2  (mod 10)

Why must p be prime? Because a prime modulus guarantees that every nonzero
element has a multiplicative inverse. If p were composite (say p = 12), then
some elements (like 4) would have no inverse: there is no x such that
4·x ≡ 1 (mod 12). With a prime p, every nonzero a has a unique inverse
a⁻¹ such that a·a⁻¹ ≡ 1 (mod p), computed via Fermat's little theorem:
a⁻¹ = a^{p-2} mod p.

## secp256k1: The Curve

Bitcoin uses the elliptic curve secp256k1, defined by the equation:

    y² = x³ + 7  (mod P)

where P is the specific 256-bit prime:

    P = 2²⁵⁶ - 2³² - 977

This is the *base field* prime. Point coordinates (x, y) live in F_p: both x
and y are integers in {0, 1, ..., P-1}, and they must satisfy the curve
equation modulo P.

See `constants.py` for the exact value:

```python
from frost.constants import P
# P = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEFFFFFC2F
```

## Points on the Curve

A point on secp256k1 is a pair (x, y) satisfying y² ≡ x³ + 7 (mod P). Not
every (x, y) pair is a valid point: most aren't. The set of valid points,
together with a special "point at infinity," forms a mathematical group.

```python
from frost.point import Point

# Construct a point (validates it's on the curve)
p = Point(x_value, y_value)  # raises ValueError if not on curve

# The point at infinity (identity element)
identity = Point()  # x=None, y=None
```

## Point Addition

Two points on the curve can be "added" to produce a third point on the curve.
Geometrically, you draw a line through the two points, find where it
intersects the curve again, and reflect across the x-axis. Algebraically,
this reduces to a formula involving the coordinates and modular arithmetic.

Key properties of point addition:
- **Closure**: the sum of two curve points is always another curve point
- **Associativity**: (A + B) + C = A + (B + C)
- **Identity**: P + O = P (where O is the point at infinity)
- **Inverse**: P + (-P) = O (negation reflects y: -P = (x, P - y))
- **Commutativity**: A + B = B + A

The implementation lives in `Point.__add__` and `Point._dbl` (for the special
case where both points are the same, which uses the tangent line instead of a
secant).

## Scalar Multiplication

Scalar multiplication is repeated point addition: k·G means "add G to itself
k times." This is the core operation in elliptic curve cryptography.

```python
from frost.point import G

# 42 times the generator point
result = 42 * G
```

The critical property: given k and G, computing k·G is fast (256 doublings
and at most 256 additions). But given k·G and G, recovering k is
computationally infeasible. This is the *elliptic curve discrete logarithm
problem*, and it's the foundation of all security in FROST and BIP340.

The implementation uses the double-and-add algorithm (see `Point.__rmul__`),
with a precomputed table optimization for the generator point G (see
`_fast_ge_mul` in `point.py`).

## The Generator Point G

G is a specific point on secp256k1 that generates the entire group: every
point on the curve can be expressed as k·G for some integer k. The exact
coordinates are defined in the secp256k1 specification and hardcoded in
`constants.py`.

```python
from frost.point import G

# G is a specific point with known coordinates
print(G.x)  # 0x79BE667EF9DCBBAC55A06295CE870B07...
```

## The Scalar Field Z_Q

The group generated by G has a specific finite order: there are exactly Q
points, where Q is another large prime:

    Q ≈ 2²⁵⁶ (slightly less than P)

This means Q·G = O (the point at infinity). Scalar multiplication "wraps
around" after Q steps, so scalars are naturally elements of Z_Q: integers
modulo Q.

Private keys, nonces, secret shares, Lagrange coefficients, and signature
components are all scalars in Z_Q. The `Scalar` class encapsulates this:

```python
from frost.scalar import Scalar

a = Scalar(42)
b = Scalar(17)
c = a + b        # addition mod Q
d = a * b        # multiplication mod Q
e = a.inv()      # modular inverse: a · e ≡ 1 (mod Q)
```

## Two Fields, Two Purposes

This is a common source of confusion: secp256k1 involves *two* distinct
fields, and they serve different purposes.

| Field | Modulus | What lives here | Where it appears |
|-------|---------|----------------|-----------------|
| F_p | P (base field prime) | Point coordinates (x, y) | Curve equation, point arithmetic |
| Z_Q | Q (group order) | Scalars (private keys, nonces, shares) | Scalar multiplication, FROST protocol |

P and Q are different primes (both ~2²⁵⁶ but not equal). When you do
arithmetic on *coordinates* (inside point addition), you reduce mod P. When
you do arithmetic on *scalars* (private keys, shares, nonces), you reduce
mod Q.

The `Scalar` class handles Z_Q arithmetic. The `Point` class handles F_p
arithmetic internally. As a user of these classes, you rarely need to think
about P directly, but understanding that these are two separate fields is
essential.

## The Point at Infinity

The point at infinity O is the identity element for point addition:
P + O = P for any point P. It plays the role that zero plays in ordinary
addition.

It arises naturally when you add a point to its negation: P + (-P) = O.
In the `Point` class, the point at infinity is represented by x = y = None:

```python
identity = Point()       # point at infinity
assert identity.is_zero()
```

## x-only Public Keys (BIP340)

BIP340 introduces a convention: public keys are represented by their
x-coordinate alone (32 bytes instead of 33). For any x-coordinate on the
curve, there are exactly two valid y values (one even, one odd). BIP340
always chooses the *even* y.

This convention simplifies serialization and reduces on-chain data. The
`lift_x` function recovers the full point from an x-coordinate:

```python
# Recover the even-y point from an x-coordinate
p = Point.lift_x(x_value)
assert p.has_even_y()
```

Throughout FROST, public keys and nonce commitments may need to be "negated"
(switching from even-y to odd-y or vice versa) to maintain BIP340
compatibility. You'll see this in the signing module.

## Summary

| Concept | Code | Purpose |
|---------|------|---------|
| Scalar (Z_Q element) | `Scalar(42)` | Private keys, nonces, shares |
| Point on curve | `Point(x, y)` | Public keys, commitments |
| Generator point | `G` | Base point for scalar multiplication |
| Scalar multiplication | `k * G` | Derive public key from private key |
| Point at infinity | `Point()` | Identity element for addition |
| x-only key | `Point.lift_x(x)` | BIP340 public key convention |

With scalars and points understood, we can now build Schnorr signatures.
